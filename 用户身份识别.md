# wechatauto 轻量身份解析与自动首绑功能改造文档

## 一、目标

为当前 `wechatauto` 项目增加一套“轻量身份解析层”，用于解决以下问题：

1. 当前系统更偏向按会话处理，而不是按真实用户处理
2. 用户改昵称、改备注、改群昵称后，历史记忆和画像容易断裂
3. 纯手动首绑过于繁琐，不适合后续扩展成长期运行的 Windows 应用
4. 完全依赖大模型自动绑定身份风险过高，容易误归并

本次改造的目标不是构建一个重型 CRM，而是实现：

- Agent 自动生成初始身份草稿
- 规则优先完成身份解析
- 中间态支持低成本人工确认
- 后续逐步过渡到稳定的 `resolved_user_id`

---

## 二、核心设计思想

### 2.1 不再要求完整人工首绑
新用户首次出现时，不要求人工填写完整用户画像。

改为：
- 系统自动提取基础身份信息
- Agent 自动分析用户意图和关系
- 自动生成一个“身份草稿”
- 只在必要时让人工确认

### 2.2 不直接做重型身份系统
当前阶段不要一上来就做：
- 大而全的用户主数据平台
- 复杂跨会话身份图谱
- 全自动高风险归并

当前阶段只做：
- 可运行
- 可回溯
- 可扩展
- 可渐进增强

### 2.3 规则优先，大模型辅助
身份绑定优先使用低成本、可解释、可控的规则：

- 别名命中
- 最近会话连续性
- 群关系上下文
- 历史关键词重合
- 当前消息意图与用户类型匹配

大模型只负责：
- 草拟身份标签
- 提取初始画像字段
- 给出候选解释
- 提供辅助判断

大模型不直接做最终唯一身份拍板。

### 2.4 先解决 80% 场景
先覆盖最常见场景：
- 新用户首次进来
- 用户改昵称
- 群聊中同一人更换群昵称
- 疑似旧用户再次出现

暂不追求复杂跨设备、多账号、强实名级别身份识别。

---

## 三、推荐实现路线（比原方案更经济）

## 路线结论

不推荐：
- 纯手动首绑
- 首次就完整录入用户画像
- 一开始就把所有身份字段都做全
- 完全依赖大模型自动合并

推荐：
- 轻量自动首绑
- 小字段草拟
- 别名表 + 草稿用户表
- 规则优先 + 人工兜底

---

## 四、分阶段方案

## Phase 1：最小可用身份层（推荐优先实现）
目标：尽快让系统“先认人，再回消息”。

### 功能
1. 新增 `resolved_user_id`
2. 新增轻量身份草稿表
3. 新增别名表
4. 新消息到来时做身份解析
5. 规则命中则绑定旧用户
6. 无命中则生成 `draft user`
7. agent 自动抽取少量身份字段

### 只抽取这些字段
- canonical_name（建议称呼）
- user_type（客户/朋友/同事/供应商/未知）
- current_intent（咨询/售后/催办/闲聊/合作）
- relationship_to_me（把我当客服/老板/同学/朋友等）
- identity_confidence（置信度）

### 优点
- 成本低
- 改动小
- 很适合接到当前脚本架构里
- 不会把系统搞得太重

---

## Phase 2：半自动修正
目标：支持用户改昵称、改群昵称后的修正。

### 功能
1. 历史 aliases 命中
2. 当前消息与历史用户画像做轻量匹配
3. 生成候选合并建议
4. 中等置信度进入审核队列
5. 高置信度自动追加 alias

### 匹配信号
- 当前 display_name 是否接近历史 alias
- 是否来自相同群
- 是否延续最近对话话题
- 消息关键词是否接近该用户历史主题
- 用户对我的称呼是否一致

---

## Phase 3：记忆分层
目标：把“人”和“会话”彻底拆开。

### 功能
1. 用户记忆：按 `resolved_user_id`
2. 会话记忆：按 `conversation_id`
3. 回复生成时同时读取
   - 单聊：优先用户记忆
   - 群聊：用户记忆 + 群记忆

### 说明
这一阶段再替换当前按 `chat_id` 存记忆的方式。

---

## 五、为什么这个方案更经济

### 5.1 比纯手动首绑便宜
不需要每个用户都手工完整录入画像，只在必要时确认。

### 5.2 比完整身份图谱轻
不需要一开始就做复杂数据库和管理后台，用本地 JSON 存储即可。

### 5.3 比完全模型驱动更稳
规则优先，可解释，可控，误判成本更低。

### 5.4 更适合你现在的项目阶段
当前项目还是脚本驱动、运行时数据本地存储，因此更适合先加一层轻量 identity 内核，而不是重做系统。

## 七、消息模型改造

扩展当前消息对象，增加以下字段：

```
resolved_user_id: str | None
conversation_id: str | None
participant_display_name: str | None
relationship_to_me: str | None
current_intent: str | None
identity_confidence: float | None
identity_status: str | None  # resolved / draft / review / unknown
identity_evidence: list[str] | None
```

要求：

- 保留旧字段兼容老逻辑
- 新逻辑优先使用 `resolved_user_id`
- 没有 `resolved_user_id` 时允许降级回旧逻辑

## 八、新增模块

建议新增目录：

```
wechat_ai/
  identity/
    __init__.py
    identity_models.py
    identity_repository.py
    alias_manager.py
    identity_matcher.py
    identity_bootstrapper.py
    identity_resolver.py
```

### 模块职责

#### identity_models.py

定义：

- User
- DraftUser
- UserAlias
- IdentityCandidate
- IdentityResolutionResult

#### identity_repository.py

负责读写：

- users.json
- user_aliases.json
- draft_users.json
- identity_candidates.json

#### alias_manager.py

负责：

- 查询 alias 是否命中
- 追加 alias
- 更新 latest seen name

#### identity_matcher.py

负责规则评分：

- alias exact match
- name similarity
- same group context
- keyword overlap
- recent context continuity

#### identity_bootstrapper.py

负责：

- 用 agent 生成初始身份草稿
- 提取用户类型、当前意图、关系判断
- 输出 DraftUser

#### identity_resolver.py

总入口：

- 先查 alias
- 再查规则候选
- 再决定自动合并 / 生成草稿 / 进入审核

## 九、处理流程

## 9.1 新用户消息到来

1. 提取原始身份信息
2. 查询 alias
3. 若命中：
   - 返回旧 `user_id`
4. 若未命中：
   - 调用 `identity_bootstrapper`
   - 生成 `draft user`
   - 返回临时 `identity_status=draft`

## 9.2 疑似旧用户改名

1. alias 未命中

2. 调用 `identity_matcher`

3. 根据规则打分：

   - > = 0.85：自动绑定旧用户并追加 alias

   - 0.60 ~ 0.85：写入 candidate，等待审核

   - < 0.60：作为新 draft user

## 9.3 回复生成

回复链路优先读取：

1. `resolved_user_id` 对应用户信息
2. 若没有，则读取 draft user 的轻量画像
3. 若仍没有，则走旧逻辑兜底

------

## 十、Agent 在这个系统里的职责

Agent 只负责“草拟”，不直接做最终身份裁决。

### Agent 输出内容

- 建议称呼
- 用户类型
- 当前意图
- 对我的关系判断
- 简短原因
- 置信度

### Agent 不负责

- 直接唯一合并旧用户
- 覆盖已有 confirmed user
- 直接修改历史主键

------

## 十一、接入点

## 11.1 wechat_runtime.py

在构造 Message 前，加入：

```
identity_result = identity_resolver.resolve(raw_message)
```

把结果写入 Message。

## 11.2 reply_pipeline.py

画像加载顺序修改为：

1. resolved_user_id
2. draft user profile
3. 旧逻辑兜底

## 11.3 memory

第一阶段不强制改写旧记忆结构，只保留接口兼容。
 第二阶段再引入：

- user memory
- conversation memory

## 十三、验收标准

### P0 验收

1. 新用户可自动生成 draft identity
2. alias 命中时可绑定到旧用户
3. 用户改昵称后可自动追加新 alias
4. 回复链路可以读取 draft 或 resolved 身份
5. 不破坏现有自动回复流程

### P1 验收

1. 能生成 candidate review
2. 能区分自动合并与待确认
3. 身份日志可追踪

### P2 验收

1. 用户记忆与会话记忆分层
2. GUI 审核接口预留完成

------

## 十四、明确不做的事

当前版本不要做：

- 全自动高风险身份合并
- 重型数据库
- 复杂后台管理系统
- 跨微信账号统一身份图谱
- 过度依赖大模型自由判断

------

## 十五、开发要求

1. 必须增量开发，不推翻现有结构
2. 必须兼容当前脚本入口
3. 必须优先复用现有日志、画像、记忆、RAG 机制
4. 必须把身份系统做成独立模块
5. 必须做到规则优先、模型辅助
6. 必须允许后续演进到 Windows GUI

------

## 十六、最终产出目标

实现一个适合当前项目阶段的身份解析内核：

- 新用户自动草拟身份
- 老用户通过 alias 和规则重新识别
- 模型负责理解，规则负责约束
- 少量人工确认即可完成长期可维护的身份绑定

这个版本的目标不是“完美识别所有人”，而是：
 **用最低成本，把当前项目从‘按会话回复’推进到‘按用户回复’。**
