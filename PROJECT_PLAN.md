# WeChat AI 运行时项目计划

## 项目定位

这个仓库最初是一个 Windows 微信桌面自动化项目，现在在自动化层之上加入了一套结构化的 `wechat_ai` 运行时。

当前项目目标已经不只是“调模型生成一句回复”，而是要把整条自动回复链路做成可调试、可配置、可扩展的运行时系统，核心关注点包括：

- 消息接入与发送
- 提示词编排
- 用户画像与助手画像
- 本地知识检索
- 可观测性与调试
- 轻量记忆

如果你想看模块和运行时结构，请读 [架构总览](/C:/github/pywechat/pywechat-main/pywechat-main/docs/architecture-overview.md)。这份文档只负责讲阶段状态、路线图和剩余收尾工作。

## 已完成的阶段

增强版运行时目前已经按几个阶段逐步搭起来：

1. 基础重构
   把简单回复流程拆成 `wechat_ai` 下可复用的运行时模块。
2. 画像与配置
   增加用户画像、助手画像、默认值和集中配置入口。
3. 提示词流水线
   增加消息标准化、上下文管理、提示词构建和回复编排。
4. RAG 基础能力
   增加本地知识导入、分块、嵌入抽象、检索和重建索引脚本。
5. 可观测性与记忆
   增加结构化日志、排障查看脚本、记忆存储、记忆摘要注入和会话快照写回。

历史实现过程记录保存在 [docs/superpowers/plans](/C:/github/pywechat/pywechat-main/pywechat-main/docs/superpowers/plans) 和 [docs/superpowers/specs](/C:/github/pywechat/pywechat-main/pywechat-main/docs/superpowers/specs) 中。它们仍然有价值，但现在属于“演进记录”，不再是项目主入口。

## 当前状态

### 当前已经具备

- `wechat_ai/wechat_runtime.py`
  作为运行时入口协调器，连接微信自动化层与 AI 模块
- `wechat_ai/orchestration/`
  提供消息解析、上下文管理、提示词构建和回复流水线
- `wechat_ai/profile/`
  提供用户画像、助手画像及其 JSON 存储
- `wechat_ai/rag/`
  提供本地知识导入、分块、规范化、检索与 rerank 抽象
- `wechat_ai/memory/`
  提供文件型轻量记忆与会话快照
- `wechat_ai/logging_utils.py`
  提供结构化 JSONL 日志能力
- `scripts/show_recent_logs.py` 与 `scripts/show_memory_summary.py`
  提供第一线排障工具
- 运行时模块已经有单测、脚本测试和 smoke 覆盖

### 当前运行时能做到什么

- 对单个好友自动回复
- 对群聊中 `@` 消息自动回复
- 轮询全局未读会话
- 把用户画像和助手画像注入提示词
- 把本地知识检索结果注入提示词
- 在有记忆摘要时将其注入提示词
- 记录运行时结构化事件
- 在生成回复后追加会话快照到记忆存储

## 完成度判断

这个项目已经不再是零散原型，当前已经具备：

- 比较清晰的运行时模块边界
- 本地持久化的数据目录
- 可直接用于排障的日志和记忆查看路径
- 针对主要运行时模块的非零测试面

但它更适合被定义为：

“一个已经成型、可继续扩展的本地 WeChat AI runtime MVP”

而不是：

“已经完全适合长期无人值守运行的成熟系统”

## 离长期使用还差什么

### 1. 运行时稳定性还需要更强

- 需要更多真实 Windows 微信长时间运行验证
- 需要更强的 UI 漂移、窗口异常、消息形态异常处理
- 需要更明确的发送失败、导航失败、恢复失败时的回退策略

### 2. 运维层需要补强

- `runtime_events.jsonl` 需要日志轮转或保留策略
- memory snapshot 需要保留策略和裁剪策略
- `wechat_ai/data/` 需要更明确的备份、迁移和恢复说明

### 3. 记忆能力还只是轻量版

- 目前记忆摘要仍然是手动或占位驱动
- 还没有自动摘要更新逻辑
- 还没有短期工作记忆与长期稳定偏好的更清晰分层

### 4. 知识库能力还需要运维化

- 还缺少更明确的知识源管理流程
- 目前检索后端仍然偏本地开发态
- 检索结果的解释性、排序质量和引用查看还有提升空间

### 5. 生产安全能力不足

- 还缺少更完整的密钥管理说明
- 还缺少内容安全、敏感信息处理和回复边界规则
- 还缺少人工接管、暂停、限流等高风险场景控制能力

### 6. 操作体验还可以更完善

- 还缺少画像编辑/管理脚本
- 还缺少真正的运行手册和故障恢复手册
- 还缺少更紧凑的日志/记忆运维视图

## 最后的收尾清单

如果目标是“接近可长期使用”，建议把下面这份清单当作最后收尾范围：

- [ ] 增加完整运行手册：环境准备、启动、停止、排障、恢复
- [ ] 增加画像管理脚本或更明确的运维入口
- [ ] 增加真实微信场景下的 live smoke 验证
- [ ] 增加日志轮转或日志保留策略
- [ ] 增加记忆裁剪与保留策略
- [ ] 增加 UI 自动化异常下的更安全 fallback 和恢复策略
- [ ] 将 fake embeddings 替换或扩展为更真实的检索后端选项
- [ ] 增加更明确的密钥管理和敏感信息处理规范
- [ ] 增加生产场景下的内容安全规则
- [ ] 增加 `wechat_ai/data/` 的数据归属、备份与迁移说明

## 推荐的下一步顺序

如果下一目标是“更稳定、更适合长期跑”，建议优先顺序如下：

1. 运行手册与画像运维工具
2. 真实场景 smoke 与失败恢复增强
3. 日志与记忆的保留控制
4. 更强的检索后端与知识管理流程
5. 生产安全控制能力
