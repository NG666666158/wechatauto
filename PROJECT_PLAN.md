# 项目计划与当前进度

## 项目当前所处阶段

这个项目已经完成了“从单点自动回复脚本到可扩展运行时”的核心过渡。

现在的状态更适合定义为：

`后端核心基本可用，桌面客户端产品化进行中。`

可以把整体进度粗略理解为：

- 后端运行时主链路：`80% ~ 85%`
- 真实微信守护与联调能力：`70% ~ 80%`
- 桌面客户端服务层：`65% ~ 75%`
- 前后端完整产品化闭环：`50% ~ 60%`
- 文档与运维体系：`65% ~ 75%`

## 已完成

### 1. 微信自动回复主链路

- 单聊自动回复
- 群聊 `@` 自动回复
- 全局轮询未读会话
- 多条未读消息聚合回复
- 未读顺序重排与回归测试

### 2. 编排与提示词链路

- `Message` 标准化
- 上下文裁剪
- Prompt 构建
- Reply pipeline
- provider 调用边界整理

### 3. 身份与画像

- 用户身份识别
- identity draft / candidate 管理
- 用户画像读写
- 助手画像读写
- 分层自我身份
  - 全局
  - 关系
  - 用户覆盖

### 4. 知识库与检索

- 本地知识库导入
- 索引重建
- 文档切块
- 本地检索
- 联网扩库基础能力
- 支持 `txt/md/json/pdf/docx/图片 OCR`

### 5. 记忆与可观测性

- 结构化 JSONL 日志
- memory store
- 会话快照写回
- 日志查看脚本
- memory 查看脚本
- 守护模式心跳与 backoff

### 6. 桌面客户端后端壳层

- `DesktopAppService`
- 设置读写
- daemon 状态管理
- 客户列表与身份草稿聚合
- 自我身份管理接口
- 知识库导入接口

## 已完成但仍需真实联调

### 1. 真实微信启动链路

已经有：

- 首次启动引导脚本
- 讲述人接入
- 正式监听脚本切换

但还需要继续验证：

- 不同窗口状态下的一致性
- 首启和日常启动切换体验
- 更稳定的停止与恢复行为

### 2. 真实微信运行稳定性

已经解决过的问题包括：

- WMI 检测兜底
- UTF-8 控制台输出
- 未读消息顺序
- 部分群 sender 透传

仍需继续观察：

- 长时间运行下的 UI 漂移
- 特殊消息形态
- 微信窗口切换与焦点异常

## 尚未完成

### 1. 桌面客户端完整闭环

还缺：

- 前端页面与后端服务层完整接线
- 桌面壳层集成
- 托盘、静默后台、定时启停的完整产品化体验

### 2. 会话页真实能力闭环

`DesktopAppService` 中仍有部分接口没有真正接到微信运行时，例如：

- `suggest_reply(...)`
- `send_reply(...)`

它们现在仍偏向桌面页预留接口，而不是完整业务闭环。

### 3. 运维与长期运行能力

还需要继续补：

- 日志轮转与保留策略
- memory 裁剪与保留策略
- 失败恢复策略说明
- 备份与迁移说明
- 更完整的运行手册

### 4. 安全边界

还需要继续补：

- 更明确的敏感信息处理规则
- 更稳的人工接管策略
- 高风险消息的保护边界

## 下一阶段重点

接下来建议只围绕两条主线推进：

### 主线一：桌面客户端开发

- 明确客户端页面与后端接口映射
- 补齐前端交互状态
- 接通客户端设置、客户页、知识库页、消息页

### 主线二：真实运行稳定性

- 做更完整的 live smoke
- 继续压实启动/停止/恢复
- 补长期守护与异常回退策略

## 最后一轮收尾清单

如果目标是“接近可长期使用”，建议以这份清单为准：

- [ ] 补完整桌面客户端开发文档
- [ ] 完成前后端接口接线
- [ ] 补齐消息页真实回复与人工接管闭环
- [ ] 增加日志轮转与清理策略
- [ ] 增加 memory 裁剪与保留策略
- [ ] 增加数据备份、迁移、恢复说明
- [ ] 增加更完整的真实微信长时间 smoke
- [ ] 增加更清晰的安全规则与高风险场景保护
